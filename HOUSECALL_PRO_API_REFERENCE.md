## Product/Access Reality

Housecall Pro API access is not universal across all plans. The help center says Public API/API key access is available to Max plan Pros. API keys are generated by Admin users from the Housecall Pro App Store API app card.

For `hcp`, this means Sprint 1 must start with credential feasibility:

- Confirm the client is on Max or has API access enabled.
- Confirm whether the key is a Company API Key or Application API Key.
- Confirm whether multi-location access is required.
- Confirm whether we have a true sandbox or a live test company.
- Do not assume OAuth unless Rozeta becomes/uses an official integration partner app.

## Authentication

Housecall Pro supports API keys and OAuth 2.0.

API key auth is the normal path for a Pro/custom integration:

```http
Authorization: Token <api-key>
```

OAuth is for official integration partners:

```http
Authorization: Bearer <access-token>
```

OAuth endpoints:

- Authorization URL: `https://pro.housecallpro.com/oauth/authorize`
- Token/refresh URL: `https://api.housecallpro.com/oauth/token`
- OAuth flow: Authorization Code Flow
- OAuth credentials are requested from `apideveloper@housecallpro.com`

OpenAPI security schemes:

- `Company API Key`: `Authorization` header, `Token <api-key>`
- `Application API Key`: `Authorization` header, `Token <api-key>`
- `Housecall User OAuth Token`: OAuth 2.0

Implementation guidance for `hcp`:

- V1 should implement API key auth first.
- Store auth mode explicitly: `api_key` or `oauth`.
- For API key mode, build the header as `Authorization: Token <key>`.
- For OAuth mode, build the header as `Authorization: Bearer <token>`.
- Do not log tokens.
- `hcp auth doctor` should verify base URL, auth mode, token presence, and a low-risk request like `GET /company` or `GET /application`.

## Request Conventions

Base URL:

```text
https://api.housecallpro.com
```

Common request headers:

```http
Accept: application/json
Authorization: Token <api-key>
Content-Type: application/json
```

Common pagination:

- Most list endpoints use `page` and `page_size`.
- Some endpoints use `page` and `per_page`.
- Default `page` is usually `1`.
- Default `page_size` is usually `10`.
- Paginated responses commonly include `page`, `page_size`, `total_pages`, and `total_items`.

Array query parameters:

- The docs model many filters as arrays: `employee_ids`, `location_ids`, `work_status`, `status`, `tag_ids`, etc.
- Implement encoding with repeated bracket params where accepted by the API, for example `employee_ids[]=...`.
- Validate against live API behavior in `ENG-253` because docs may not show exact serialization for every endpoint.

Multi-location:

- Many list endpoints accept `location_ids`.
- The docs say `location_ids` are ignored when `X-Company-Id` is set.
- The changelog added `X-Company-Id` support on 2025-10-23.
- `hcp` should eventually expose `--location-id` and a config default for multi-location Pros.

Date/time:

- Scheduling fields use ISO-8601 style date/time strings.
- Job/estimate scheduled filters use `scheduled_start_min`, `scheduled_start_max`, `scheduled_end_min`, and `scheduled_end_max`.
- Invoice filters use `created_at_*`, `due_at_*`, and `paid_at_*`.

## Implementation Risks and Unknowns

- Rate limits are not clearly documented in the public pages I found. Treat 429 handling and retry/backoff as required anyway.
- Some operations in the OpenAPI export do not repeat security metadata even though the API is authenticated. Do not treat missing `security` as public access.
- Docs distinguish Company API Key and Application API Key, but a Pro-generated key workflow may not make that distinction obvious. `ENG-253` must verify key type with real credentials.
- Some endpoint families use `/api/...` prefixes while most core endpoints do not. Do not normalize paths blindly.
- Work status filter values and response labels differ in formatting. Normalize statuses in our store.
- Attachments use multipart/binary upload after a documented breaking change. Keep attachment upload out of reporting V1.
- Mutating endpoints should require `--dry-run` or explicit confirmation before we expose them in the CLI.

## Direct Fit to `hcp` Commands

Owner brief:

- Leads: `GET /leads`
- Booked appointments: `GET /jobs/{job_id}/appointments`, `GET /jobs?expand=appointments`
- Estimates: `GET /estimates`
- Revenue booked/completed: `GET /jobs`
- Unpaid invoices: `GET /invoices`
- Top techs: jobs plus assigned employees
- Marketing source performance: leads/jobs/estimates grouped by `lead_source`

Leads commands:

- `hcp leads today`: `GET /leads` sorted/filtered by dates after local sync
- `hcp leads stale`: local derived metric from lead status, created/updated timestamps, assigned employee, and conversion state
- `hcp leads by-source`: `GET /leads` plus `GET /lead_sources`

Funnel:

- Lead: `GET /leads`
- Appointment: `GET /jobs/{job_id}/appointments`
- Estimate: `GET /estimates`
- Sold job: `GET /jobs`
- Completed job: `GET /jobs` with completed status
- Paid invoice: `GET /invoices` with paid status

Sales:

- `hcp estimates unsold`: `GET /estimates`, estimate options and statuses
- `hcp estimates no-followup`: estimates plus option notes/activity if available
- `hcp sales leaderboard`: estimates/jobs grouped by assigned employee

Fulfillment:

- `hcp jobs active`: `GET /jobs` with active statuses
- `hcp jobs stalled`: local derived metric from job status, schedule, timestamps, invoices, notes, and appointments
- `hcp jobs missing-invoice`: jobs compared to `GET /jobs/{job_id}/invoices` and `GET /invoices`

Technician:

- Employees: `GET /employees`
- Completed jobs per tech: `GET /jobs`
- Revenue per tech: jobs/invoices grouped by assigned employees
- Checklist compliance: `GET /checklists`

Marketing:

- Sources: `GET /lead_sources`
- Leads by source: `GET /leads`
- Revenue by source: jobs/estimates/invoices joined back to lead source

Cash:

- Open/overdue/aging: `GET /invoices`
- Job invoice detail: `GET /jobs/{job_id}/invoices`

Automation/MCP:

- Start read-only with local-store report commands.
- Delay writes, webhooks, lead conversion, schedule updates, and delivery sends until dry-run behavior is built.
